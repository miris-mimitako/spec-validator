# spec-validator

## 目的

このリポジトリは、仕様書・設計・実装・テストのあいだにある不整合を検出し、
一貫性監査とルールバリデーションを実行するためのスキル定義を蓄積する。

最終的には、以下を継続的に検証できることを目標とする。

- 仕様書に書かれた要求が、実装対象として十分に定義されていること
- 実装が仕様から逸脱していないこと
- テストが仕様と実装の両方を適切にカバーしていること
- レビュー時に確認すべきルールが明文化され、再利用可能であること

## このリポジトリで定義していくもの

- 仕様レビューの観点
- 設計レビューの観点
- 実装レビューの観点
- テストレビューの観点
- 各成果物間のトレーサビリティ確認ルール
- ルール違反を検出するための監査手順
- 将来的に自動化するための入力形式・判定基準・出力形式

## 基本方針

- 単なるチェックリストではなく、判定可能なルールとして記述する
- 人間のレビューにも、自動バリデーションにも使える粒度を目指す
- 曖昧な表現を避け、入力・判定条件・期待結果を明確にする
- 仕様書からテストまでを分断せず、一連の流れとして扱う

## 命名規則方針

### 基本原則

- 命名は DDD のユビキタス言語を優先する
- 名前は実装都合ではなく、ドメイン上の意味を表す
- 同じ概念に複数の名前を使わない
- ドメイン概念と技術概念を名前の上で混在させない

### DDD を標準とする対象

以下は、原則としてドメイン用語で命名する。

- Entity
- Value Object
- Aggregate
- Domain Service
- Repository
- Factory
- Domain Event
- Use Case / Application Service
- 仕様書上の業務概念に対応する型・関数・モジュール

### 言語・フレームワーク標準との関係

DDD を優先するが、言語やフレームワークに強い慣習がある箇所では、その標準語彙を尊重する。
ただし、その場合でもドメインの意味が失われてはならない。

判断原則は以下の通りとする。

- ドメイン概念を表す名前は DDD の用語で決める
- 技術的な役割を表す接尾辞・接頭辞は、言語やフレームワークの標準に合わせてもよい
- フレームワークが予約している責務名や一般慣習名は無理に置き換えない
- ただし、フレームワーク都合の名前だけでドメイン概念を表現してはならない

### 優先順位

命名で迷った場合の優先順位は以下とする。

1. 仕様書・業務用語として定義された名称
2. DDD のモデルとして一貫する名称
3. 言語・フレームワークの標準命名
4. チーム独自の略語や慣習

### 許容する命名の考え方

- `Order` や `CustomerId` のようなドメイン名は優先して維持する
- `Repository` `Controller` `DTO` `Entity` など、技術的責務を示す名称は標準慣習に従ってよい
- ただし `DataManager` `Info`, `Util`, `Helper` のような意味の弱い汎用名は避ける
- 略語は、仕様書またはチームで意味が固定されている場合に限って使う

### 命名レビューで確認する観点

- その名前は仕様書の用語と対応しているか
- その名前から責務と境界が推測できるか
- ドメイン概念なのか技術概念なのかが判別できるか
- 同じ概念に別名が混在していないか
- フレームワーク都合の名前に引きずられて、ドメイン意味が薄れていないか

## 命名規則の機械判定原則

このリポジトリで定義する命名規則は、Python による機械判定を前提とする。
そのため、レビュー観点レベルの曖昧な表現ではなく、原則として文字列パターンで判定可能な形式に落とす。

NestJS を採用する場合の標準語彙と DDD 補完語彙の正本は、
`_docs/_GroundRules/00_standard-language-ddd.yaml` に定義する。

機械判定ルールは、以下を満たさなければならない。

- 判定対象が明確であること
- 許可パターンと禁止パターンが明確であること
- 例外条件が列挙可能であること
- 人間の解釈ではなく、入力文字列から再現可能に判定できること

### 機械判定用の命名分類

命名対象は、少なくとも以下のいずれかに分類して判定する。

- Domain Term
- Application Term
- Infrastructure Term
- Framework Term
- Test Term

この分類に属さない曖昧な命名ルールは定義しない。

### 命名ルールの記述形式

命名ルールは、今後原則として以下の形式で定義する。

- ルールID
- カテゴリ
- 判定対象
- 許可パターン
- 禁止パターン
- 例外
- 判定ロジックの要点
- 違反メッセージ

### 厳格運用の原則

- `意味が弱い名前を避ける` のような曖昧な規則はそのまま採用しない
- `禁止語一覧` `許可接尾辞一覧` `必須接尾辞一覧` のように列挙型へ分解する
- `文脈に応じて判断する` という規則は、機械判定対象から除外するか、別途メタデータ入力を必須にする
- 例外は自由記述にせず、明示的な whitelist として管理する

## 命名規則案 001: DDD 優先と標準語彙の境界

### ルールID

- NAMING-001

### 目的

- ドメイン名と技術責務名を分離し、Python で一貫して判定可能にする

### 対象

- クラス名
- モジュール名
- 関数名
- ファイル名

### 判定条件

- ドメイン概念を表す主語は、仕様書または用語集に存在する名称を使う
- 技術責務を表す語は、許可された接尾辞または接頭辞のみ使用できる
- ドメイン概念を表さない汎用語のみで構成された名前を禁止する

### 許可する技術責務語

機械判定上、以下の語を技術責務語として許可する。

- `Entity`
- `ValueObject`
- `Aggregate`
- `Repository`
- `Factory`
- `DomainService`
- `ApplicationService`
- `UseCase`
- `Command`
- `Query`
- `Dto`
- `Mapper`
- `Controller`
- `Gateway`
- `Client`
- `Adapter`
- `Policy`
- `Specification`
- `Test`
- `Fixture`

### 禁止する汎用語

機械判定上、以下の語を単独使用または主要語として使用することを禁止する。

- `Manager`
- `Helper`
- `Util`
- `Utils`
- `Common`
- `Info`
- `Data`
- `Item`
- `Object`
- `Base`
- `Impl`
- `Temp`
- `Misc`

### 例外

- フレームワークや標準ライブラリが要求する固定名
- 既存の外部 API 名と一致させる必要がある名前
- テストダブルとして慣習的に使う `Stub` `Mock` `Fake` `Spy`

### 期待する出力

- 違反時は、対象名、違反語、違反種別、許可候補を返す

## 命名規則案 002: 技術責務語は補助語であり主語ではない

### ルールID

- NAMING-002

### 目的

- `Repository` や `Controller` のような技術語だけで名前が構成されることを防ぐ

### 対象

- クラス名
- ファイル名

### 判定条件

- 許可された技術責務語を使う場合、その前または後にドメイン語を含まなければならない
- 技術責務語のみで構成された名前を禁止する

### 違反例

- `Repository`
- `UserDataManager`
- `CommonHelper`
- `BaseController`

### 許容例

- `OrderRepository`
- `CreateOrderUseCase`
- `CustomerQueryService`
- `BillingPolicy`

## 命名規則案 003: 例外は定義ファイルで管理する

### ルールID

- NAMING-003

### 目的

- 判定不能な曖昧例外を防ぎ、Python 実装で再現可能にする

### 対象

- すべての命名ルール

### 判定条件

- 例外対象はコード中コメントではなく、例外定義ファイルに列挙されていなければならない
- 例外定義に存在しないものは例外として扱わない

### 備考

- 将来的には `naming_exceptions.yml` のような外部定義を前提にする
